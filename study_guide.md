# LyfSync 2.0: Architectural & Codebase Study Guide

This guide breaks down the advanced concepts in LyfSync 2.0 into plain English. It is designed to prepare you for job interview questions about system design, AI engineering, and database choices.

---

## 1. The Big Picture (System Design)

LyfSync 2.0 is an **NLP-First, RAG-grounded nutrition tracker**. 

### The Problem It Solves:
Standard nutrition apps make users search database lists or take photos. Photo apps fail on complex cuisines (like Indian cooking) because they cannot see hidden oils, ghee, or the exact weight of cooked preparations. 

### Our Solution:
The user describes what they ate in plain text (e.g., *"2 paneer parathas and a bowl of curd"*).
1. We segment the text into distinct dishes.
2. We search a local database using **Vector Search (Embeddings)** to find matching food items or pre-computed recipes.
3. We use the LLM to parse quantities and units, and automatically decompose complex dishes if they aren't pre-saved.
4. We convert units (like "bowl") to grams and lookup the final macros.

---

## 2. Database Schema & Vector Search (RAG)

Instead of using a basic relational database, we use **PostgreSQL with the `pgvector` extension** to perform semantic vector searches.

### The Tables:
1.  **`Meal`**: Stores the final logged meal with the total macros (calories, protein, carbs, fat) and the user's raw text.
2.  **`USDARaw`**: Standard base ingredients (USDA FoodData Central) containing nutritional info per 100g. Each description has a **1536-dimensional vector embedding** generated by OpenAI's `text-embedding-3-small` model.
3.  **`ICMRRaw`**: Indian regional food items (Indian Council of Medical Research), also embedded for semantic search.
4.  **`Staple`**: Pre-saved recipes for complex cooked dishes (e.g., *"Dal Tadka"*, *"Chicken Biryani"*). It stores the ingredients and their weights.
5.  **`StaplesReview`**: A holding queue for new cooked dishes that the AI is reviewing to generate recipes for.
6.  **`ReferenceServing`**: A dictionary that maps units to weights (e.g., *"1 medium apple"* -> 182g, *"1 slice bread"* -> 30g).

### What is Vector Search (pgvector)?
Instead of matching exact keywords (which fails if a user types *"spiced yogurt"* but the database has *"curd"*), we translate the text into a vector (a list of 1536 numbers representing semantic meaning). We then ask PostgreSQL to find the row whose vector is closest to our input.

```sql
-- What pgvector runs under the hood:
SELECT id, description, cosine_distance(embedding, :input_embedding) AS distance 
FROM usda_raw 
ORDER BY distance LIMIT 1;
```

## 3. The 5-Stage Meal Pipeline (`MealPipeline` Class)

Every time a user logs a meal, it flows through the `MealPipeline` class in six sequential stages:

```
[User Text Input]
       │
       ▼
 1. EXTRACT       --> Uses gpt-4o-mini to split text into distinct items.
       │
       ▼
 2. GROUND        --> Generates embeddings and queries pgvector DB tables.
       │              Finds closest match (Staples first, then ICMR/USDA).
       │
       ▼
 3. PORTIONS      --> Uses LLM to parse quantity/unit. Decomposes unmatched
       │              cooked dishes into raw ingredients.
       │
       ▼
 4. CONVERT       --> Converts units (e.g., '1 cup') into grams using 
       │              ReferenceServing table.
       │
       ▼
 5. NUTRITION     --> Looks up the final macros from the database using 
       │              the calculated gram weights.
       │
       ▼
 6. PERSIST       --> Saves the Meal to DB and spawns background tasks 
                      to generate recipes for new cooked items.
```

---

## 4. Advanced AI Engineering: The LLM Jury

When a user logs a cooked dish that **does not exist** in our database (e.g., *"Moms chicken curry"*), the pipeline cannot find a template. 

To solve this, we spin up an **Asynchronous LLM Jury** in the background:

1.  **Parallel Generation:** We query `gpt-4o-mini` **three times in parallel** with different temperatures (creativity settings) to draft 3 independent recipes for the dish. We do this concurrently using Python's `asyncio.gather` so it is extremely fast.
2.  **The Judge:** We feed all 3 draft recipes into a fourth LLM call (the "Judge"). The Judge reviews the drafts, removes outliers, and synthesizes a single consensus "golden baseline" recipe.
3.  **Review Queue:** The golden baseline is saved into `StaplesReview` for database admin approval, showing a highly mature database-seeding workflow.

## 💡 Top 3 Interview Questions you can now answer:

1.  **"Why use a multi-stage pipeline instead of one single LLM call?"**
    *   *Answer:* A single LLM call has to do too much reasoning: split text, match items to a database, do unit conversion math, and calculate macros. This leads to hallucinations. By breaking it into steps, we use database lookups for math and data, and only use the LLM for translation and parsing.
2.  **"What is pgvector and why did you use it?"**
    *   *Answer:* Pgvector is a PostgreSQL extension for vector similarity search. We use it to perform semantic searches on food names (e.g. matching "rotis" to "chapati") which standard keyword search (`LIKE %text%`) cannot handle.
3.  **"Why run the LLM Jury in a background task?"**
    *   *Answer:* Calling the LLM 4 times to generate a recipe consensus takes 3–5 seconds. If we did this synchronously during the API request, the user would see a slow, lagging app. By using FastAPI's `BackgroundTasks`, we instantly return the meal log to the user and generate the recipe template asynchronously.
